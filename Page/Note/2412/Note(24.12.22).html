HTTPS 通信 Handshake


TLS1.2 + TCP 基盤 HTTPS

TCP Handshake → TLS Handshake

Client → Server：SYN

S → C：SYN-ACK

C → S：ACK

C → S：ClientHello

S → C：ServerHello

C → S：Fin

S → C：Fin

Req/Res


TLS1.3 + TCP 基盤 HTTPS (TLS Handshake 過程が１往復短縮)

TCP Handshake → TLS Handshake

Client → Server：SYN

S → C：SYN-ACK

C → S：ACK

C → S：ClientHello

S → C：ServerHello, Fin

C → S：Fin, Req


TLS1.3 + 0RTT TCP 基盤 HTTPS (既存セッションがある場合、Session Resumptionを通じて0RTTを行う)

TCP Handshake → TLS Handshake

Client → Server：SYN

S → C：SYN-ACK

C → S：ACK

C → S：ClientHello, Req(前のセッション情報からの情報を送り、Reqを同時に転送)

S → C：ServerHello, Res, Fin(サーバーから受け取った前のセッション情報を検証し、問題がなければ受諾してResを転送)

C → S：Fin


TLS + QUIC 基盤 HTTPS (TLS HandshakeをQUIC初期設定と同時に行う)

Client → Server：QUIC(TLS：ClientHelloを含む)

S → C：QUIC(TLS：ServerHello, Fin)

C → S：QUIC(TLS：Fin)

Req/Res


TLS + 0RTT QUIC 基盤 HTTPS　(既存セッションがある場合、Session Resumptionを通じて0RTTを行う)

Client → Server：QUIC(TLS：ClientHelloを含む), Req (前のセッション情報からの情報を送り、Reqを同時に転送)

S → C：QUIC(TLS：ServerHello, Fin), Res (サーバーから受け取った前のセッション情報を検証し、問題がなければ受諾してResを転送)

C → S：QUIC(TLS：Fin)


要約

TLS1.2: Handshake段階が多く、初期設定に時間を取られる。

TLS1.3：Handshake段階が省かれており、1.2より効率的。

QUIC：QUICの長所+HandshakeとTLS Handshakeを同時に処理するので早い。

0RTT：リコネクション時にHandshakeを省いてデータ転送を開始するので早い。



Process&Thread

プロセスはOS上で実行される一つの独立したプログラム。プロセスは専用メモリ空間とシステム資源を持つ。

スレッドは一つのプロセス内で実行される作業の単位。一つのプロセスは複数のスレッドを持ち、スレッドは基本的にメモリを共有する。

ProcessとThreadの概念以外にも、CPUからのスケジューリング方法についても語っていた。以前CS/OS関連の講義を見ていた時に学んだ内容が(MutexやCOWなど)ざっくりとした感じで出てきたので、案外理解はできた。
