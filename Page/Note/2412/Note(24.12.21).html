gRPC

gRPCはGoogleが開発したRemote Procedure Call Frameworkとして、相違するシステム間の通信をより効率的かつ簡単に処理するために設計された。


主な特徴

- HTTP/2を基盤としたMultiplexing・Header圧縮、Server-Pushを提供する。

- RESTとは違いJSONではなくProtocol Buffersを活用し高速かつ効率的なserialization・Deserializationを行う。

- 多様な言語と互換性を持つ(クライアントとサーバーを違う言語で具現加)

- 両方向ストリーミングが出来る(ReqとResが同時にストリーミングできる)。

- Schema基盤設計(Protoファイルをもって明示的Interfaceを定義する)。

WebRTC (Web Real-Time Communication)

WebRTCはブラウザー間Real Time Communicationを支援するOpensource技術。

主に音声、映像、データ転送のためのProtocolとAPIを提供し、Plug-inや追加ソフト無しでWeb RTCを実現する。


主な特徴

- Plug-in無しのブラウザー基盤のRTC

- P2P通信; 中間サーバーを通さずクライアント間の直接通信 → サーバーの負荷を減らし、遅延を最小化する。

- SSL/TLS暗号化を基本的に提供し、安全な通信を保障


主な構成要素

- MediaStream; 外部入力装置からデータを取り、他Peerへ転送を行う。

- RTCPeerConnection; 他Peerとのコネクションを設定、維持、データを交換するAPIで、ビデオや音声ストリームを転送したり、ICE・STUN・TURNを使ってNATとFirewallを通れるように手伝う。

- RTCDataChannel; Peer間のデータをリアルタイムで転送するAPI。


コネクション過程

1．クライアントがgetUserMedia APIを通じて自身のメディア情報を要請。

2．SDP(Session Description Protocol)を通じてセッション情報交換。
クライアントは自身のメディア情報とコネクションできる候補IPを含んだSDPを作り交換する。

3．ICE(Interactive Connectivity Establishment)を使って両クライアントがコネクションできる最適の経路を探す。

4．Peerコネクション・データ転送。


だが、一番の恐れはNATやFirewallの問題である。

NATとFirewall環境によっては直接的な通信が難しい場合があるので、TURNサーバーを使用した中継が必要となる → コスト増加、遅延
