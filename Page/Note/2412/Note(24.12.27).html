Database Indexing


TableとIndexがどのようにDiskに書き込まれるのか。

基本的にはExcelの表のような形で、当たり前だがRowとColumnデータとして書き込まれる。

それを読み込む時の作業が大事なわけで、Table情報はPage単位で一定大きさごとに束ねられて書き込まれ、読み込む時もPage単位で参照しながら目標を検索する。

そのため、膨大なTableから特定目標を検索するとき、Index無しで作業をするとなるとそのTableを0Pageから参照して目標が出るまで作業を繰り返してしまう(Full Table Scan)。 するとデータ参照が無駄に多くなり処理時間も増加するため、適切にIndexを設定することでそれを塞ぐことができる。

Non-Clustered IndexはTable情報をもってIndex Key、PKを基準で新しいPageを作り、実際のデータではないPointerを書き込む。 なので検索を行う時はそのPointerを基にPKのIndexから目標を探索して結果を読み込む方式を取る。

Clustered Indexは、上記のようにPK Indexは別途に管理されるため、基本的にPKがClustered Indexと見ていい。Indexがもう既に整列されてる場合使える。 その理由はPKはデータが書き込まれる際、PKの順によって書き込み位置が決められるからだ。なので、UUIDといったランダムな文字列をPKとして使うのは効率が悪い。 なので、ランダムな文字列を使用しなければならない場合はClustered Indexを使わないか、UUIDを順次的に生成されるように設計することを考慮すべき。


Row based table & Column based table


言葉通り、データをRow基盤で書き込むかそれともColumn基盤で書き込むかによって区分される。

Row based tableは一般的なDBの形であり、ユーザーデータ管理やTransaction処理などに使われるが、 Column based tableの場合はデータ分析などの特定作業が行われるアプリで採用されているDB管理方法。

書き込み方はあまり難しく考えることはなく、下のようにデータがColumn単位で書き込まれると思えばそれでいい。


1, 2, 3 (ID)

a, b, c (name)

4, 4, 5 (value1)

5, 5, 4 (value2)


例えば、value1やvalue2だけの合計や平均が欲しい場合、必要なRowだけ読み込めば作業が終わるのでとても効率的。

だが、ID 1のすべての情報が欲しい場合は多数のメモリに接近して読み込みを行わなければならないのでとても効率が悪い(書き込み、修正、削除も)。

この場合は一般的なRow based tableの方が一回だけですべての情報を読み込めるので効率的。

つまり、要約すると、データ分析を主に行うアプリケーションの場合はColumn based tableのほうが効率的だと言える。
