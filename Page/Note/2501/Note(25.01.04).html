B-TREE B+TREE

Original B-tree

B-treeはTree構造のNodeの集合体で構成されている。

TableのIDと、そのPageに移動するためのTIDがKeyとValueとして記されている。

親ノードを基準に探しているIDが親ノード要素のKeyより小さいか大きいかでどの子ノードに移動するのかを決める。なのでFull table scanより効率的で速い。


だがすべてのNodeがKeyとValueを持ってなければならないのでメモリの浪費になることもある。そしてそのせいで不必要なIO作業が行われ遅延が発生する。あとは範囲queryを実行する時に(例えばID１～５までの読み込み)Treeを数回巡回することになるので効率が悪い。

B+tree

Original B-treeの問題点を補完して最適化された。Leaf NodeだけTIDを持つ。そしてそのLeaf Nodeは順番にLinkedされているので、範囲Queryにも効率的に作動する。Leaf Node以外はIDだけを索引に使うために持っていて、Leaf NodeのKeyと被るようになっているが、最小限のコスト。


Database Partitioning

Tableを任意の単位で分割して管理する方法。100億個のRowがあるTableより、1億のRowのTableで作業を行う方が明らかに早い。Partitioning方法にはVertical・Horizontalの二つの方法がある。

Partitioning基準としては日付・IDなどの範囲だったり、使用者の地域・年齢などの集計だったり使用環境上での任意で決める。 その中にはHash functionによるPartitioningもある。

Vertical Partitioning

言葉通り、垂直分割。Rowを分割して(Column単位)複数の塊にする。例えばサイズの大きいBlob Fieldがあり、それを分割したりという用途で使う。

Horizontal Partitioning

言葉通り、水平分割。Row全体を一つの塊として範囲ごと分割する方法(1~200Row、201~400Row)。

Partitioning/Sharding

Partitioningは単一DB内でTableを物理的に分割して管理し、DB性能を引き上げる技術。

Shardingは複数のDB Serverに分散セーブして、トラフィック制御の機能も含めている技術。
